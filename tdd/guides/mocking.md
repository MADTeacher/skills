# Mock, fake и stub

## Главное правило

Подменяй внешние границы системы. Не мокай внутренние детали собственного кода.

Мок внутренних классов часто превращает тест в проверку реализации.

## Внешние границы, которые можно подменять

Обычно можно использовать mock, fake или stub для:

- сторонних API;
- платежных провайдеров;
- email/SMS/push-сервисов;
- очередей сообщений;
- файловой системы;
- системного времени;
- генераторов случайности;
- сетевых клиентов;
- базы данных, если реальная или тестовая база нецелесообразна;
- медленных, нестабильных или дорогих зависимостей.

## Что не стоит мокать

Обычно не нужно мокать:

- собственные доменные сервисы;
- внутренние use case классы;
- приватные методы;
- мапперы, если они часть контролируемого кода;
- валидаторы, если они часть поведения;
- репозитории, если можно использовать in-memory реализацию или тестовую базу;
- внутреннюю оркестрацию.

## Mock

Mock проверяет взаимодействие: был ли вызов, с какими параметрами, сколько раз.

Используй mock осторожно. Он подходит, когда сам факт внешнего вызова является наблюдаемым контрактом.

Пример:

```pseudocode
test "sends notification after order is confirmed" {
  notifications = mock(NotificationGateway)
  orders = createOrderService(notifications: notifications)

  orders.confirm(orderId)

  verify(notifications.sent("order_confirmed", to: customerId))
}
```

Это допустимо, если отправка уведомления — внешний эффект, который и нужно проверить.

## Fake

Fake — рабочая упрощенная реализация зависимости.

Пример:

```pseudocode
class InMemoryUserStore implements UserStore {
  users = []

  save(user) {
    users.add(user)
  }

  findByEmail(email) {
    return users.find(user => user.email == email)
  }
}
```

Fake часто лучше mock, потому что:

- тест ближе к реальному поведению;
- меньше утверждений о внутренних вызовах;
- проще рефакторить реализацию;
- можно тестировать состояние и результат.

## Stub

Stub возвращает заранее заданный ответ.

Пример:

```pseudocode
class AlwaysSuccessfulPaymentGateway implements PaymentGateway {
  charge(amount) {
    return PaymentResult.success(id: "payment-test-id")
  }
}
```

Stub хорош для внешних зависимостей, когда нужно смоделировать конкретный ответ.

## Как выбрать

```text
Нужно проверить внешний эффект вызова? → mock
Нужна простая рабочая замена зависимости? → fake
Нужен заранее заданный ответ? → stub
Можно использовать реальную зависимость безопасно и быстро? → используй реальную
```

## Предупреждающие сигналы

Тест, вероятно, слишком привязан к реализации, если:

- почти все зависимости — mock;
- основная часть теста — verify вызовов;
- тест ломается при перестановке внутренних шагов;
- тест не проверяет конечный результат;
- чтобы понять тест, нужно знать внутреннее устройство модуля.
