# Рефакторинг после GREEN

## Главное правило

Рефакторинг выполняется только после зеленых тестов.

Не рефактори в состоянии RED. Если тест красный, сначала сделай минимальную реализацию, которая возвращает систему в GREEN.

## Цель рефакторинга

Рефакторинг должен улучшать дизайн без изменения поведения.

Возможные цели:

- убрать дублирование;
- сделать имена точнее;
- выделить value object;
- уменьшить публичный интерфейс;
- переместить логику ближе к данным;
- углубить модуль;
- упростить ветвления;
- отделить внешнюю границу от доменной логики;
- улучшить читаемость тестов.

## Безопасный процесс

```text
1. Убедись, что тесты зеленые.
2. Сделай маленькое изменение.
3. Запусти релевантные тесты.
4. Если тесты зеленые, продолжай.
5. Если тесты красные, откати или исправь немедленно.
```

## Что искать после GREEN

Проверь:

- появилась ли дублирующаяся логика;
- не слишком ли много параметров у функции;
- не протекли ли детали реализации в тесты;
- не пора ли выделить доменный результат;
- не используется ли примитив там, где нужен value object;
- не смешались ли валидация, бизнес-логика и внешние вызовы;
- не появился ли мелкий модуль, который ничего не инкапсулирует;
- не скрывает ли название метода реальное доменное действие.

## Пример рефакторинга через value object

До:

```pseudocode
registerUser(email: String)
```

После:

```pseudocode
registerUser(email: EmailAddress)
```

Если тесты написаны через поведение, а не через внутренние шаги, такой рефакторинг будет безопаснее.

## Когда не рефакторить

Не рефактори, если:

- тесты красные;
- непонятно, какое поведение должно сохраниться;
- изменение слишком крупное для одного шага;
- рефакторинг смешивается с новой функциональностью;
- нет тестового покрытия вокруг изменяемого поведения.

Сначала добавь минимальный поведенческий тест, затем рефактори.
