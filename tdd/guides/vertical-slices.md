# Вертикальные срезы

## Смысл

В TDD лучше двигаться тонкими вертикальными срезами, а не горизонтальными слоями.

Вертикальный срез — это маленький, но законченный путь через систему, который дает наблюдаемое поведение.

Он может проходить через:

- входной интерфейс;
- валидацию;
- бизнес-логику;
- хранение данных;
- интеграцию;
- UI или API;
- тесты.

Но срез должен оставаться узким.

## Горизонтальная декомпозиция — плохо

```text
1. Создать таблицы
2. Реализовать репозитории
3. Реализовать сервисы
4. Реализовать API
5. Реализовать UI
6. Написать тесты
```

Проблемы:

- долго нет проверяемого поведения;
- архитектура проектируется до обратной связи от тестов;
- ошибки интеграции обнаруживаются поздно;
- тесты пишутся после реализации и часто повторяют ее;
- сложно понять, что уже работает.

## Вертикальная декомпозиция — хорошо

```text
1. Пользователь может создать минимальную сущность с обязательными полями
2. Пользователь видит созданную сущность в списке
3. Пользователь не может создать сущность с невалидными данными
4. Пользователь может обновить одно разрешенное поле
5. Пользователь видит понятную ошибку при конфликте версий
```

Каждый пункт можно проверить самостоятельно.

## Один TDD-цикл на один срез

```text
RED
  Тест на один сценарий.

GREEN
  Минимальная реализация этого сценария.

REFACTOR
  Улучшение структуры без изменения поведения.
```

Затем следующий срез.

## Как выбрать следующий срез

Приоритет обычно такой:

1. Самый критичный happy path.
2. Самый вероятный или дорогой failure path.
3. Валидация и ограничения.
4. Авторизация и доступность данных.
5. Идемпотентность, конкуренция, повторные действия.
6. Интеграционные крайние случаи.
7. Наблюдаемость и эксплуатационные сценарии.

## Размер среза

Срез слишком большой, если:

- для одного теста нужно реализовать много несвязанных правил;
- непонятно, почему тест падает;
- требуется изменить слишком много компонентов сразу;
- нельзя объяснить результат одной фразой;
- его нельзя принять или продемонстрировать отдельно.

Срез слишком маленький, если:

- он меняет только внутренний метод;
- после него нет наблюдаемого поведения;
- его нельзя проверить без знания реализации;
- он не приближает пользователя к полезному сценарию.
