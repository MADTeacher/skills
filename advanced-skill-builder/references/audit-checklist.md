# Audit Checklist

Используй этот файл при review/audit существующего навыка и перед финальной
сдачей сложной правки.

## Audit Mode

Когда пользователь просит review/audit skill:

1. Читай `SKILL.md` как исполняемую инструкцию.
2. Проверь frontmatter и trigger surface.
3. Проверь, может ли агент выполнить первый типовой сценарий.
4. Проверь, нет ли лишнего отчета, README-текста или истории анализа.
5. Проверь routing: каждый resource должен иметь причину.
6. Проверь layer coherence: повторы, конфликты, shadowed triggers.
7. Проверь workflow bypass: optional wording, validation bypass, script
   bypass, required resources marked optional.
8. Проверь validation: что именно запускается или смотрится.
9. Проверь failure modes и fallback.
10. Дай список правок по приоритету.

Критика должна быть жесткой: если документ нельзя использовать как навык,
скажи это прямо и предложи переписать структуру.

## Anti-Patterns

### Отчет под видом навыка

Признаки:

- дата анализа;
- область анализа;
- краткий вывод по репозиторию;
- пофайловые таблицы;
- история появления;
- длинные списки наблюдений без инструкций.

Исправление: вынеси анализ в notes, оставь в skill только поведение.

### Лишние ресурсы без сценария

Признаки:

- `assets/`, `scripts/`, `references/`, `demos/` созданы без маршрутов;
- README описывает больше, чем сам skill;
- demo существует только для солидности.

Исправление: удали или отложи все, что не нужно первым сценариям.

### Слабый trigger

Признаки:

- description слишком общий;
- реальные пользовательские формулировки не перечислены;
- условия применения описаны только в теле.

Исправление: перепиши `description` как trigger surface.

### Монолитный `SKILL.md`

Признаки:

- длинные справочники внутри основного файла;
- много вариантов, API и troubleshooting без маршрутизации;
- агент должен прочитать все, чтобы выполнить простую задачу.

Исправление: оставь core workflow, вынеси детали в routed references.

### Конфликтующие слои

Признаки:

- `SKILL.md` задает одно правило, reference или script help задает другое;
- один и тот же default, формат или термин называется по-разному;
- подробность в reference меняет core workflow вместо того, чтобы дополнять его;
- условия срабатывания спрятаны в теле или references, а не в `description`.

Исправление: выбери canonical место, убери дубль, добавь ссылку или
decision tree, а при реальном конфликте явно укажи приоритет.

### Обход обязательного workflow

Признаки:

- validation обязателен, но fallback разрешает пропустить его;
- deterministic script можно заменить ручным best-effort без blocker;
- required schema/reference описан как optional;
- критичный шаг помечен словами `если нужно`, `можно`, `желательно`;
- fallback не возвращает агента к основному workflow и не сообщает риск.

Исправление: явно укажи condition, action, required resource/script,
validation и blocker/fallback для обязательного шага.

### Форматные обещания без pipeline

Признаки:

- skill обещает PDF/PPTX/MP4/JSON/XML/export и другие форматы;
- нет scripts;
- нет constraints;
- нет проверки результата.

Исправление: добавь pipeline или убери обещание.

## Final Checklist

Перед сдачей навыка проверь:

- [ ] Навык создан или обновлен в `.agents/skills/<skill-name>/SKILL.md` либо
  в явно указанном пользователем пути.
- [ ] `SKILL.md` начинается с YAML frontmatter.
- [ ] Во frontmatter есть только `name` и `description`.
- [ ] `description` содержит реальные trigger phrases.
- [ ] Тело можно использовать как инструкцию агенту.
- [ ] Нет пофайлового отчета, даты анализа и истории создания.
- [ ] Workflow достаточно конкретен для первого сценария.
- [ ] Каждый resource имеет маршрут использования.
- [ ] Верхний слой кратко резюмирует или маршрутизирует, а не пересказывает
  нижний слой целиком.
- [ ] References дополняют, а не переопределяют `SKILL.md`.
- [ ] Термины, defaults и tool choices едины во всех слоях.
- [ ] Нет конфликтующих `must/never` между `SKILL.md`, references и scripts.
- [ ] Обязательные шаги не помечены как optional wording без условия.
- [ ] Validation нельзя пропустить через fallback без blocker или waiver.
- [ ] Required references/scripts не заменяются ручным best-effort без риска.
- [ ] Нет обязательных README/demos/assets/scripts без причины.
- [ ] Форматные обещания поддержаны scripts или явно ограничены.
- [ ] Есть validation protocol.
- [ ] Для сложного навыка есть план forward-testing.
