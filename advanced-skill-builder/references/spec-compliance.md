# Проверка спецификации

Используй этот файл как канонический справочник по Agent Skills specification
при создании, правке, аудите и финальной сдаче навыков.

## Источник правил

- Навык - это директория с обязательным `SKILL.md`.
- `SKILL.md` содержит YAML frontmatter и Markdown-инструкции.
- Минимальный frontmatter содержит `name` и `description`.
- Дополнительные стандартные поля: `license`, `compatibility`, `metadata`,
  `allowed-tools`.
- `scripts/`, `references/`, `assets/` - необязательные ресурсы. У каждого
  должен быть маршрут из `SKILL.md`.
- Ссылки на файлы пиши относительно корня навыка. Избегай глубоких цепочек
  ссылок между references.

## Строгая проверка

Для навыков, которые ты создаешь или правишь, применяй строгий режим:

- `name`: строка длиной 1-64 символа; только `a-z`, `0-9`, `-`; без начального,
  конечного и двойного дефиса; значение совпадает с именем папки навыка.
- `description`: строка длиной 1-1024 символа; непустая; говорит, что делает
  навык и когда его использовать; содержит конкретные trigger words.
- `license`: строка с коротким названием лицензии или ссылкой на bundled
  license file, если поле нужно.
- `compatibility`: строка длиной 1-500 символов, если у навыка есть реальные
  требования к среде, продукту, пакетам или сети.
- `metadata`: mapping со строковыми ключами и строковыми значениями; ключи лучше
  делать уникальными для клиента или автора.
- `allowed-tools`: строка со списком tools через пробел; поле
  экспериментальное, его поддержка зависит от agent runtime.

Если строгая проверка падает, исправь skill до записи или перед финальной сдачей.
Не сдавай такой skill как проверенный.

## Проверка совместимости

При аудите чужих навыков можно разделить ошибки на блокеры и предупреждения:

- блокер: нет `SKILL.md`, YAML нельзя разобрать, нет `name`, нет
  `description`, `description` пустой.
- блокер для создания или правки: нарушение лимитов `name`, `description`,
  `compatibility`, неверный тип стандартного поля, несовпадение `name` с папкой.
- предупреждение для совместимости между клиентами: клиент с мягкой проверкой
  может загрузить часть навыков с косметическими нарушениями, но при создании
  или правке их нужно исправлять.
- предупреждение: `allowed-tools` задан, но runtime не обещает поддержку этого
  поля.

В ответе различай: «валидно по спецификации», «может загрузиться клиентом с
мягкой проверкой и предупреждением», «нельзя считать проверенным навыком».

## Постепенная загрузка

`description` загружается на старте для всех навыков, поэтому держи ее короткой
и точной. Она должна быть до 1024 символов.

`SKILL.md` загружается целиком после активации. Держи его до 500 строк и
примерно до 5000 токенов. Если подробности раздувают файл, вынеси их в
`references/`, `scripts/` или `assets/` и добавь точный маршрут чтения.

Ресурс полезен только тогда, когда агент знает, когда его открыть или запустить.
Фраза «смотри references» слишком слабая для progressive disclosure.

## Протокол проверки

Перед финальной сдачей навыка:

1. Запусти `uv run scripts/validate-skill.py <skill-dir>` из директории
   `advanced-skill-builder`, если скрипт доступен.
2. Проверь routing ресурсов: каждый reference, script и asset достижим из
   `SKILL.md`.
3. Проверь layer coherence: подробности в references дополняют `SKILL.md`, но
   не меняют базовые правила.
4. Проверь trigger surface: условия применения находятся в `description`, а не
   прячутся только в теле или references.
5. Если автоматическая проверка недоступна, назови блокер, выполни ручную
   проверку и явно укажи риск.
