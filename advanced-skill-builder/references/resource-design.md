# Resource Design

Этот файл раскрывает выбор степени свободы, progressive disclosure и bundled
resources. `SKILL.md` остается canonical источником core workflow.

## Degrees of Freedom

Подбери форму инструкции под хрупкость задачи:

| Степень свободы | Использовать когда | Форма |
|---|---|---|
| Высокая | Есть много допустимых решений, важны эвристики и judgment | Короткие правила, критерии, примеры |
| Средняя | Есть предпочтительный паттерн, но нужны варианты | Псевдокод, decision tree, templates |
| Низкая | Операция хрупкая, повторяемая, легко сломать формат | Script, фиксированная последовательность, smoke test |

Чем дороже ошибка, тем меньше свободы оставляй агенту.

## Progressive Disclosure

`SKILL.md` держи коротким. Детали выноси в ресурсы только если они
действительно нужны.

Используй:

- `references/` для доменных правил, schemas, API, policies,
  troubleshooting, длинных примеров;
- `scripts/` для детерминированных, повторяемых или хрупких операций;
- `assets/` для файлов, которые используются в итоговом артефакте: шаблонов,
  изображений, шрифтов, boilerplate, fixtures;
- `test-prompts.json` или fixtures для регрессии поведения, если навык
  достаточно сложный;
- `agents/openai.yaml` только как UI-facing metadata, если среда навыков это
  поддерживает и оно реально нужно.

Правило маршрута: каждый ресурс должен иметь в `SKILL.md` явное правило, когда
его читать, запускать или использовать.

## Before Creating a Resource

Перед созданием каждой папки спроси себя:

- Будет ли агент реально использовать это в первом или втором типовом
  сценарии?
- Уменьшает ли ресурс повторяемую работу?
- Предотвращает ли ресурс дорогую ошибку?
- Можно ли проверить, что ресурс работает?
- Есть ли в `SKILL.md` маршрут к этому ресурсу?

Если ответов нет, ресурс не добавляй.

## Resource Matrix

| Ресурс | Добавлять когда | Не добавлять когда | Обязательная проверка |
|---|---|---|---|
| `references/workflow.md` | Workflow длиннее 1 экрана | Процесс помещается в `SKILL.md` | В `SKILL.md` есть маршрут чтения |
| `references/schema.md` | Есть схемы данных/API/форматов | Схема нестабильна или неизвестна | Пример использования схемы |
| `references/policy.md` | Есть доменные правила или запреты | Правило одно и короткое | Не конфликтует с workflow |
| `references/troubleshooting.md` | Есть повторяющиеся сбои | Сбоев еще не наблюдали | Симптомы и действия конкретны |
| `scripts/verify.*` | Результат можно автоматически проверить | Проверка только субъективная | Скрипт запускался |
| `scripts/export.*` | Формат является обещанием навыка | Формат может пригодиться | Есть preflight и smoke test |
| `scripts/transform.*` | Операция хрупкая или длинная | Однократная простая команда | Есть input/output пример |
| `assets/templates/` | Нужен reusable стартовый материал | Шаблон будет мешать вариативности | Есть инструкция применения |
| `assets/examples/` | Примеры нужны как fixtures | Примеры только декоративные | Есть ожидаемый результат |
| `test-prompts.json` | Навык сложный и требует регрессии | 2 примера достаточно в `SKILL.md` | Проверяемые criteria |
| `demos/` | Качество видно только в артефакте | Демо будет витриной без пользы | Демо открывается или рендерится |
| README | Предназначен для людей, чтобы они ознакомились с возможностями агентского навыка | Навык внутренний и агентский | README не дублирует `SKILL.md` |

## Layered Instruction Coherence

Перед финальной записью или аудитом собери instruction-bearing файлы:
`SKILL.md`, routed `references/`, `scripts/*` usage/help, agent metadata и
assets/templates, если они содержат инструкции агенту.

Найди:

- повторенные правила без ссылки на canonical место;
- несовпадающие defaults, термины, названия и версии;
- конфликтующие `must/never`, порядок действий или fallback;
- hidden “when to use” вне `description`;
- reference-детали, которые переопределяют `SKILL.md`;
- resources, не достижимые из routing.

Классифицируй находки как допустимый `summary + pointer`, лишний дубль,
конфликт, orphan detail или shadowed trigger.

## Workflow Bypass Resistance

Проверяй, нельзя ли обойти обязательную последовательность через мягкие
формулировки, fallback или ручную замену pipeline.

Каждый обязательный шаг должен иметь:

- condition: когда шаг обязателен;
- action: что именно сделать;
- required resource/script: что прочитать или запустить;
- validation: чем подтвердить результат;
- blocker/fallback: что делать, если шаг невозможен.

Ищи bypass-признаки:

- `если нужно`, `можно`, `желательно` рядом с обязательным действием;
- best-effort вместо validation;
- manual equivalent вместо deterministic script;
- required schema/reference описан как optional;
- fallback не возвращает агента к основному workflow;
- validation можно пропустить без blocker, waiver или явного риска в ответе.

Усиление: замени мягкую формулировку на `must/always` только для критичного
шага, добавь condition и validation, а невозможность выполнить шаг оформи
как blocker или explicit waiver с риском для пользователя.
