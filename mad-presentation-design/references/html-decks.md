# HTML-презентации, экспорт и проверка

HTML — рабочий исходник этого навыка. Статические экспорты строятся из HTML,
кроме редактируемого PPTX: для него с самого начала нужен HTML с ограничениями
ради корректного экспорта.

## Первое решение: формат сдачи

Спросите или выведите из задачи:

- только HTML: максимальная свобода;
- редактируемый PPTX: исходник с ограничениями, см. `pptx-authoring.md`;
- PDF: печать/экспорт из браузера, высокая визуальная точность;
- PNG: скриншоты по слайдам;
- SVG: ограниченный или «лучший возможный» путь; надежен только для простых
  нативных векторных слайдов или специально подготовленных SVG-материалов.

## Рабочая директория и Node

Для каждого проекта презентации создавайте отдельную директорию, например:

```text
my-deck/
├── package.json
├── node_modules/
├── slides/
├── exports/
└── scripts/
```

Все Node/npm-окружение, то есть среда JavaScript и ее пакетный менеджер, а также
временные скрипты агента должны жить внутри этой директории. Не устанавливайте и
не линкуйте `node_modules` в корень репозитория, в директорию навыка или в общую
рабочую папку вне конкретной презентации.

Сразу скопируйте весь каталог `scripts/` навыка в `scripts/` директории
презентации. Не копируйте отдельные файлы: у экспортов есть соседние
зависимости, например `export_deck_pptx.mjs` использует `html2pptx.js`.

Скрипты навыка запускайте из директории презентации:

```bash
node scripts/export_deck_png.mjs --slides slides --out exports/png
```

Не вызывайте экспортные скрипты по `../scripts/...` или абсолютным путям к
навыку. Презентационная папка должна быть самодостаточной: локальный Node,
локальные `node_modules`, локальные HTML-слайды, локальные копии скриптов
навыка, локальные экспорты.

## Размеры canvas

Canvas здесь означает рабочую область слайда. По умолчанию:

- HTML/PDF/PNG: `1920x1080px`;
- редактируемый PPTX: `960x540pt`, соответствует широкому формату PowerPoint.

Другие размеры используйте только по требованию площадки, шаблона или задачи.

## 2-слайдовое направление до массовой сборки

Перед созданием всей презентации сделайте:

1. обложку;
2. репрезентативный плотный контентный слайд.

Эти два слайда фиксируют визуальную грамматику: типографику, интервалы,
палитру, изображения, графики, footer, то есть нижнюю служебную строку, и
экспортные ограничения.

## Многофайловая архитектура

Многофайловая архитектура означает, что презентация разделена на несколько
файлов. Используйте ее по умолчанию для серьезных презентаций.

```text
deck/
├── index.html
├── shared/
│   ├── tokens.css
│   └── components.css
└── slides/
    ├── 01-cover.html
    ├── 02-agenda.html
    └── 03-content.html
```

Стартуйте с `assets/deck_index.html` и отредактируйте `DECK_MANIFEST`.

Плюсы:

- ошибка одного слайда не ломает всю презентацию;
- каждый слайд можно открыть отдельно;
- CSS и layout-проблемы, то есть проблемы компоновки, проще изолировать;
- разные люди могут работать над разными слайдами.

## Однофайловая архитектура

Однофайловая архитектура означает, что вся презентация лежит в одном файле.
Для маленьких презентаций и быстрых черновиков используйте `assets/deck_stage.js`.

```html
<script src="assets/deck_stage.js"></script>
<deck-stage>
  <section class="slide">...</section>
  <section class="slide">...</section>
</deck-stage>
```

Каждый `section` должен иметь фиксированные размеры и `overflow: hidden`.

## Грамматика слайда

Базовый слайд:

```html
<section class="slide">
  <p class="kicker">Раздел</p>
  <h1>Заголовок слайда</h1>
  <p class="lead">Главная мысль.</p>
  <div class="content-grid">...</div>
  <p class="slide-number">03</p>
</section>
```

Правила:

- одна главная мысль на слайд;
- основной текст обычно `24px` или больше на `1920x1080`;
- подписи обычно `18px` или больше;
- поля и номера слайдов стабильны;
- роли слайдов чередуются.

## React/Babel

Используйте React только когда он уменьшает повторение: компоненты графиков,
повторяемые макеты слайдов, canvas для вариантов или общие разделы, которые
строятся из данных. Для маленьких презентаций часто достаточно обычного HTML/CSS без
фреймворка.

Подключение:

```html
<script src="https://unpkg.com/react@18.3.1/umd/react.development.js" crossorigin="anonymous"></script>
<script src="https://unpkg.com/react-dom@18.3.1/umd/react-dom.development.js" crossorigin="anonymous"></script>
<script src="https://unpkg.com/@babel/standalone@7.29.0/babel.min.js" crossorigin="anonymous"></script>
```

Компоненты подключайте как Babel-скрипты, которые Babel переведет в обычный
JavaScript:

```html
<script type="text/babel" src="components.jsx"></script>
<script type="text/babel" src="deck.jsx"></script>
```

Каждый Babel-файл компилируется отдельно. Общие компоненты экспортируйте в
`window`, чтобы они были доступны другим файлам.

```jsx
function MetricCard({ label, value }) {
  return (
    <div className="metric-card">
      <p>{label}</p>
      <strong>{value}</strong>
    </div>
  );
}

window.MetricCard = MetricCard;
```

Держите данные слайдов в объектах, совместимых с JSON:

```js
window.DECK_DATA = {
  metrics: [
    { label: 'Revenue', value: '$12.4M' },
  ],
};
```

Используйте уникальные имена компонентов и стилей для каждой презентации. Избегайте
глобальных `styles`, `data` или `Card`, если загружается несколько скриптов.

Редактируемый PPTX читает DOM после отрисовки в браузере. Если нужен PPTX,
финальный текст должен быть в семантических текстовых тегах, а эффекты, которые
работают только в браузере, лучше исключить.

## Заметки докладчика

Используйте `aside.notes` или JSON-блок заметок. Заметки докладчика помогают
выступать, а не дублируют видимый текст.

```html
<aside class="notes">
  Перед следующим слайдом объяснить, почему изменилась метрика.
</aside>
```

## PDF-экспорт

Многофайловая презентация:

```bash
node scripts/export_deck_pdf.mjs --slides slides --out exports/deck.pdf
```

Однофайловая презентация `deck-stage`:

```bash
node scripts/export_deck_stage_pdf.mjs --html deck.html --out exports/deck.pdf
```

PDF должен сохранять визуальную точность и текст, который можно выделить и найти
поиском, если браузер может напечатать текст как вектор.

## PNG-экспорт

PNG подходит для проверки, миниатюр и статичной отправки слайдов.

```bash
node scripts/export_deck_png.mjs --slides slides --out exports/png
```

Проверьте загрузку шрифтов до скриншотов. Скрипт ждет момента, когда браузер
закончил активную работу, и `document.fonts.ready`.

PNG-экспорт всех слайдов — обязательная проверка качества перед финальной
сдачей, даже если пользователь попросил только PPTX или PDF. После экспорта:

1. Проверьте, что число PNG равно числу HTML-слайдов.
2. Просмотрите каждый PNG в полном размере, не только общий лист с миниатюрами.
3. Исправьте все обрезания, наложения, пустые картинки, съехавшие подписи,
   плохую читаемость и расхождения между слайдами.
4. Повторите экспорт PNG и проверку до чистого результата.
5. В финальном ответе назовите путь к PNG-папке и статус проверки всех слайдов.

## SVG-экспорт

SVG не является универсальным экспортом произвольного HTML. Используйте один из
путей:

- сделайте слайд или ключевое смысловое изображение как нативный SVG, если
  нужен настоящий вектор;
- экспортируйте графики/диаграммы как SVG-материалы и вставляйте их в
  HTML-презентацию;
- для произвольных HTML-слайдов сдавайте PDF/PNG и честно объясняйте, почему
  SVG будет с потерями или неремонтируемым.

## Редактируемый PPTX

Команда:

```bash
node scripts/export_deck_pptx.mjs --slides slides --out exports/deck.pptx
```

Сначала прочитайте `pptx-authoring.md`. Если исходная презентация была сделана без
ограничений для PPTX, сначала перестройте HTML в версию с такими ограничениями.

PPTX не отменяет проверку качества через PNG. Сначала добейтесь чистого
HTML/PNG-рендера всех слайдов, затем экспортируйте PPTX штатным скриптом навыка
и проверьте редактируемость текста.

## Проверка результата

Каждую HTML-презентацию нужно проверить визуально и технически до сдачи.

Чеклист:

1. HTML открывается в браузере.
2. В консоли нет релевантных ошибок.
3. Шрифты и изображения загружаются.
4. Ключевые слайды читаются в целевом размере.
5. Все слайды экспортированы в PNG через `node scripts/export_deck_png.mjs`.
6. Каждый PNG просмотрен в полном размере и все дефекты исправлены.
7. Запрошенные экспорты созданы локальными копиями штатных скриптов навыка.
8. Если был запрошен PPTX, файл создан через `node scripts/export_deck_pptx.mjs`
   и текст проверен на редактируемость.

Базовая проверка браузера:

```bash
python scripts/verify.py deck/index.html
```

Для `deck-stage` файла:

```bash
python scripts/verify.py deck.html --slides 12
```

Проверка viewport, то есть области просмотра браузера:

```bash
python scripts/verify.py deck/index.html --viewports 1920x1080,1440x900
```

Обязательная проверка качества всех слайдов через PNG:

```bash
node scripts/export_deck_png.mjs --slides slides --out exports/png
```

Ищите:

- обрезанный текст;
- плохое выравнивание;
- нечитаемые подписи графиков;
- пропавшие изображения;
- съехавшие footer, то есть нижние служебные строки, или номера;
- несовместимые отступы между слайдами.

## Проверка экспортов

Используйте только локальные копии скриптов навыка:

- `node scripts/export_deck_png.mjs` для PNG;
- `node scripts/export_deck_pdf.mjs` для PDF из многофайловой архитектуры;
- `node scripts/export_deck_stage_pdf.mjs` для PDF из однофайловой архитектуры;
- `node scripts/export_deck_pptx.mjs` для редактируемого PPTX;
- `scripts/html2pptx.js` только как низкоуровневую часть PPTX-пути навыка.

Не используйте PowerPoint, Keynote, LibreOffice, Quick Look, `open`,
`osascript`, автоматизацию рабочего стола или альтернативные самописные
экспортеры для финальных экспортов.

PDF:

- число страниц совпадает с числом слайдов;
- фоны видны;
- текст можно выделить и найти поиском, когда это ожидается;
- нет пустой страницы в конце.

PNG:

- число файлов совпадает с числом слайдов;
- изображения достаточно четкие для проверки;
- прозрачные ассеты отображаются корректно;
- каждый слайд проверен отдельно и дефекты устранены до сдачи.

SVG:

- обещайте SVG-качество только для слайдов, которые изначально сделаны как
  ограниченный вектор;
- откройте SVG хотя бы в одном просмотрщике;
- назовите растеризованные или нередактируемые части, если они есть.

PPTX:

- текст редактируется;
- изображения не пропали;
- границы слайда соблюдены;
- визуальные упрощения намеренные.

## Частые сбои

Пустой или частичный рендер:

- неверный локальный путь;
- не хватает зависимости;
- ошибка синтаксиса JSX;
- компонент не прикреплен к `window` при отдельных Babel-файлах.

Неверный шрифт:

- проверьте пути `@font-face`;
- ждите `document.fonts.ready`;
- не полагайтесь на шрифт, которого нет в среде экспорта.

Съехал layout, то есть компоновка:

- проверьте `box-sizing: border-box`;
- проверьте ширину и высоту слайда;
- посмотрите grid tracks, то есть дорожки сетки, и absolute-positioned элементы,
  то есть элементы с абсолютным позиционированием;
- проверьте, использует ли экспорт `px` или `pt`.

## Финальная сдача

Сообщайте только важное:

- какие файлы созданы;
- какие проверки запущены;
- где лежат PNG всех слайдов и что каждый PNG проверен;
- важные оговорки;
- на каких слайдах еще нужны данные пользователя.
