# Рабочий процесс: от брифа до готовой презентации

Вы — младший дизайнер презентаций. Пользователь — руководитель. Ваша задача —
сделать презентацию понятнее, убедительнее и проще для правки.

## Задавайте вопросы одним пакетом

Для новой или размытой задачи задавайте вопросы сразу списком, чтобы пользователь ответил одним сообщением. Не растягивайте бриф на цепочку одиночных вопросов, если следующий шаг не заблокирован.

Можно не спрашивать, если это маленькая последующая правка или пользователь уже
дал полный бриф, исходный текст, визуальный контекст и целевой формат.

## Жесткие правила исполнения

- Все экспорты выполняйте только штатными скриптами навыка из `scripts/`.
  Запрещено подменять их PowerPoint, Keynote, LibreOffice, Quick Look,
  `open`, `osascript`, автоматизацией рабочего стола или самописным
  экспортером.
- Не закладывайте MCP, плагины настольного браузера или инструменты, которые
  работают только внутри IDE, то есть среды разработки, как обязательную часть
  рабочего процесса.
  Переносимый путь проверки и сдачи - локальные файлы, локальная копия
  `scripts/` и обычные CLI-команды.
- Для каждой презентации создавайте отдельную рабочую директорию. Все, что
  относится к Node/npm, то есть среде JavaScript и ее пакетному менеджеру
  (`package.json`, `node_modules`, lock-файлы), временным скриптам агента,
  HTML-слайдам и экспортам, размещайте внутри этой директории.
  Не разворачивайте Node-окружение в корне репозитория или директории навыка.
- Сразу после создания директории презентации скопируйте весь каталог
  `scripts/` навыка в `scripts/` внутри презентации. Запускайте экспорт только
  из этой локальной копии: `node scripts/export_deck_png.mjs ...`. Не вызывайте
  экспортные скрипты по хрупким путям и не копируйте отдельный файл без его
  соседних зависимостей.
- Если агент создает вспомогательные скрипты для генерации данных, материалов или
  HTML, запускайте их из директории презентации. Экспортные операции все равно
  должны идти через локально скопированные скрипты навыка.
- Финальный QA-гейт, то есть обязательная проверка качества перед сдачей, нужен
  для любой презентации: экспортируйте все слайды в PNG через
  `node scripts/export_deck_png.mjs`, проверьте каждый PNG, исправьте все
  визуальные дефекты и повторяйте цикл до чистого результата.

## Обязательные группы вопросов

### 1. Цель и аудитория

- Что аудитория должна понять, решить или сделать после презентации?
- Кто аудитория и что она уже знает?
- Тон ближе к краткому докладу для руководства, обучающему докладу, продажам,
  инвесторскому питчу, внутреннему обновлению, исследованию или воркшопу?

### 2. Исходники и бренд

- Есть ли существующая презентация, структура, отчет, расшифровка, заметки, скриншоты
  или данные?
- Есть ли бренд-гайд, цвета, типографика, логотипы, изображения, скриншоты
  продукта или UI, то есть интерфейса, или примеры?
- Какие факты, цифры, имена, графики или цитаты обязательны?

### 3. Линия рассказа и объем

- Сколько нужно слайдов или сколько минут длится выступление?
- Презентация должна быть повествовательной, модульной или справочной?
- Какие разделы обязательны: обложка, повестка, проблема, решение,
  доказательства, демо-раздел, дорожная карта, бюджет, риски, следующие шаги,
  приложение?

### 4. Формат сдачи

- Какие форматы нужны: HTML, редактируемый PPTX, PDF, PNG, SVG?
- Если нужен редактируемый PPTX, можно ли сразу делать дизайн с ограничениями
  PPTX?
- Нужны ли заметки докладчика или тезисы только для выступающего?

Если нужен editable PPTX, сразу зафиксируйте, что это кроссплатформенный
PowerPoint output для Windows/macOS, а не просто HTML с последующим экспортом.

### 5. Ревью и итерации

- Первый показ нужен как 2-слайдовое направление или как более полный черновик?
- Нужны ли варианты, и по какому измерению: визуальный язык, плотность,
  структура, цвет, формулировка заголовков?
- Пользователь будет сам править презентацию после сдачи?

## Шаблон вопросов

```markdown
Перед сборкой презентации хочу зафиксировать бриф. Можно ответить одним
сообщением:

1. Аудитория и цель:
2. Число слайдов или время выступления:
3. Обязательные разделы или линия рассказа:
4. Исходные материалы и факты, которые нужно использовать:
5. Бренд-материалы или визуальные референсы:
6. Форматы сдачи: HTML / редактируемый PPTX / PDF / PNG / SVG:
7. Нужны ли заметки докладчика?
8. Сначала показать 2-слайдовое направление или собрать более полный черновик?
9. Что важнее: визуальная полировка, редактируемость, скорость или плотность данных?
10. Есть ли стили, утверждения или макеты, которых нужно избегать?
```

Для образовательных презентаций дополнительно зафиксируйте аудиторию и тон
эмоджи-подсказок: дети, подростки, взрослые, корпоративное обучение; допустимы
ли эмоджи, какие роли они должны играть и насколько крупными могут быть.

## Перед дизайном: материалы и направления

Если задача связана с конкретным брендом, продуктом, клиентом или мероприятием,
сначала выполните `brand-assets.md`:

- критичные материалы бренда, продукта и UI должны быть найдены или запрошены;
- источники и пробелы фиксируются в `brand-spec.md` внутри директории
  презентации;
- без логотипа для брендовой презентации не переходите к визуальному дизайну;
- без изображений продукта или UI для продуктовой презентации не делайте общий
  главный экран.

Если задача не брендовая, но смысл слайдов зависит от предметных изображений,
сначала выполните `content-assets.md`:

- начните со смысловой роли изображения: фотография, диаграмма, объект, сцена,
  персонаж, анатомический или учебный материал, историческое изображение, карта,
  график, текстура или
  сгенерированная иллюстрация;
- ищите локальные материалы до поиска в интернете или генерации;
- используйте реальные растровые изображения, то есть PNG/JPG/WebP, или честно
  векторные схемы, а не общую замену на CSS/SVG;
- для сложных узнаваемых объектов обязательна строгая проверка на реальное
  растровое изображение: человек,
  тело/анатомия, сцена, предмет, место, техника, еда, одежда, транспорт,
  медицинский или учебный объект должны быть PNG/JPG/WebP через `<img>`,
  сгенерированное растровое изображение или честная заглушка. Не рисуйте их
  вручную
  SVG/CSS/HTML-shapes/canvas;
- для редактируемого PPTX вставляйте значимые изображения через `<img>`, а
  подписи, словарные слова и ключевой текст оставляйте редактируемыми.

Если стиль не задан, не выбирайте "красиво" по инерции. Предложите 3
направления из `design-directions.md`:

- безопасное деловое;
- редакционное, образовательное или ведущее через данные;
- более выразительное, но уместное для аудитории.

Каждое направление должно менять линию рассказа, обложку, плотность
содержательного слайда, работу с изображениями и работу с графиками. Если
меняется только палитра, это не направление.

Если это образовательная презентация, сначала решите, нужны ли предметные
изображения, рисунки, фотографии, карты или схемы. Для английского языка,
биологии, истории, географии, начальной школы и детских уроков такая проверка
обязательна: зрителю часто нужно увидеть конкретный предмет, действие или
явление. Затем примените `content-assets.md`: подберите материал, оставьте
подписи и изучаемые слова редактируемым текстом, а стрелки и выноски сделайте
отдельными слоями. Только после этого задавайте правила использования эмодзи:
они работают как система подсказок, но не заменяют предметное изображение.
Если эмодзи уместны, выберите правила до массовой сборки: 3-5 ролей подсказок,
размерные токены, где эмодзи появляются на слайде и какие слайды обходятся без
них.

Если при этом нужен editable PPTX, не оставляйте эмодзи как редактируемый
текст: превращайте их в маленькие raster/icon assets или заменяйте словами.

## Этап 1: бриф + 2-слайдовое направление

Не бросайтесь сразу в полную презентацию. Сначала создайте или проговорите:

- цель презентации;
- предполагаемую аудиторию;
- линию рассказа;
- примерное число слайдов;
- форматы и ограничения экспорта;
- правила кроссплатформенной PPTX-безопасности, если нужен editable PPTX;
- недостающие материалы;
- `brand-spec.md`, заметки о содержательных материалах или запасное
  направление;
- обложку;
- один репрезентативный контентный слайд.

Если создаете HTML, в начале рабочего файла оставьте короткий блок
предположений:

```html
<!--
Предположения по презентации:
- Аудитория:
- Цель:
- Формат:
- Линия рассказа:
- Недостающие входные данные:
- Ограничения экспорта:
- Брендовые или содержательные материалы / запасной вариант:
- Выбранное направление:

Если направление неверное, сейчас дешевле всего его поменять.
-->
```

Обложка проверяет голос, дисциплину работы с материалами и узнаваемость.
Контентный слайд проверяет плотность, иерархию, таблицы/графики, подписи и
экспортные ограничения.

## Этап 2: система слайдов и полный черновик

После согласования направления или прямого запроса продолжать соберите систему:

- фиксированный canvas, то есть область слайда, и безопасные поля;
- `brand-spec.md` с источниками, путями к материалам, пробелами и запасными
  правилами, если презентация брендовая;
- `assets/content/` и заметки об источниках для предметных изображений, если
  презентация небрендовая, но зависит от содержательных изображений;
- типографическую шкалу для заголовка, подзаголовка, основного текста, подписи,
  выноски и подписей графиков;
- необязательную шкалу эмоджи для образовательных слайдов:
  `xs/sm/md/lg/xl`, роли и правила размещения;
- цветовые токены и нейтральную шкалу;
- повторяемые шаблоны компоновки;
- правила визуализации данных;
- стиль цитат и слайдов с доказательствами;
- разделители секций;
- нижнюю служебную строку, номер слайда и необязательные заметки.

Затем собирайте полный черновик. Чередуйте роли слайдов; не превращайте всю презентацию
в одинаковую сетку с новым текстом.

## Этап 3: презентационная полировка

Полируйте именно для презентации:

- одна главная мысль на слайд;
- текст читается в целевом размере;
- нет висячих слов и обрезанных подписей;
- выравнивание и отступы стабильны;
- нумерация и ритм секций едины;
- плотные слайды разбиты на понятные визуальные группы;
- ограничения экспорта соблюдены.
- для editable PPTX: первичные шрифты входят в безопасный whitelist;
- для editable PPTX: editable emoji text отсутствует;
- для editable PPTX: каждый HTML-слайд имеет `<meta charset="utf-8" />`;
- нет типичных шаблонных ИИ-ошибок из `design.md` и
  `design-directions.md`.
- для образовательной презентации эмоджи имеют стабильные роли, размеры и позиции, а
  не выглядят как случайный декор.

Для редактируемого PPTX еще раз проверьте каждый слайд по
`pptx-authoring.md` до экспорта.

## Этап 4: проверка и экспорт

Пройдите проверки из `html-decks.md`:

- HTML открывается в браузере;
- в консоли нет релевантных ошибок;
- все слайды экспортированы в PNG через `node scripts/export_deck_png.mjs`;
- каждый PNG просмотрен, а найденные дефекты исправлены повторным циклом;
- PDF/PNG совпадают с HTML;
- для editable PPTX font inventory, emoji-in-text и UTF-8 preflight прошли до
  записи `.pptx`;
- текст PPTX редактируется, если PPTX был запрошен;
- SVG честно назван ограниченной или «лучшей возможной» версией, если слайд не
  был сделан как нативный вектор.

## Варианты

Варианты должны помогать принять решение, а не украшать процесс.

Полезные измерения для презентаций:

- 3 направления из разных семейств `design-directions.md`;
- консервативный или редакционный визуальный язык;
- плотный слайд с данными или постепенное объяснение;
- рассказ от продукта или рассказ от проблемы;
- строгая палитра для совета директоров или более теплая обучающая палитра;
- без эмоджи, сдержанные подсказки эмоджи или крупные игровые подсказки эмоджи
  для учебных презентаций;
- обложка через изображение или типографическая обложка;
- учебный слайд с предметным изображением, схема с подписями или слайд без
  изображения, если тема объясняется только текстом и упражнением.

Используйте `assets/design_canvas.jsx` для сравнения направлений рядом.
Tweaks, то есть временные настройки, оставляйте только как HTML-инструмент
выбора параметров; не обещайте перенос Tweaks в PPTX.

## Tweaks для вариантов презентации

Tweaks — временные HTML-элементы управления для сравнения направлений
презентации. Они полезны на этапе проверки и обсуждения, но не
являются форматом сдачи и не должны обещаться внутри редактируемого PPTX.

Добавляйте Tweaks, когда:

- пользователь хочет сравнить визуальные направления;
- нужно быстро подобрать плотность, цвет или стиль графиков;
- у презентации есть несколько возможных способов построить рассказ;
- маленькая панель быстрее, чем несколько отдельных черновых файлов.

Не добавляйте Tweaks, если статичного 2-слайдового сравнения достаточно.

Полезные параметры:

- визуальное направление;
- плотность: просторная, сбалансированная или компактная;
- акцентный цвет;
- компоновка обложки;
- акцент в графиках;
- заметки докладчика: видны или скрыты;
- стиль разделителя секций.

Минимальная реализация:

```jsx
const DEFAULT_TWEAKS = {
  direction: 'executive',
  density: 'balanced',
  accent: '#2454FF',
};

function useDeckTweaks() {
  const [tweaks, setTweaks] = React.useState(() => {
    const saved = localStorage.getItem('deck:tweaks');
    return saved ? JSON.parse(saved) : DEFAULT_TWEAKS;
  });

  React.useEffect(() => {
    localStorage.setItem('deck:tweaks', JSON.stringify(tweaks));
  }, [tweaks]);

  return [tweaks, setTweaks];
}
```

Правила панели:

- панель компактная;
- панель не перекрывает содержимое слайда;
- для режимов используйте переключатели из нескольких вариантов;
- для цветов используйте цветовые образцы;
- ползунки нужны только для осмысленных числовых диапазонов;
- состояние по умолчанию выглядит как готовый дизайн.

Перед финальным экспортом выберите итоговое состояние Tweaks и зафиксируйте его
в HTML/CSS. Не оставляйте финальную презентацию зависимой от живой панели управления,
если пользователь явно не просит инструмент просмотра только в HTML.

## Когда не хватает входных данных

- Не хватает фактов: спросите или проверьте, не выдумывайте.
- Нет бренда: явно назовите запасное направление.
- Нет критичного логотипа, материала продукта или UI для брендовой презентации:
  остановитесь и запросите материал; продолжайте только после явного согласия на
  запасной вариант.
- Нет данных: поставьте заметную заглушку и перечислите, что нужно.
- Инструкции конфликтуют: назовите конфликт и выберите наименее рискованный
  путь или спросите, если выбор заметно меняет результат.

## Стиль финальной сдачи

Коротко:

```markdown
Готово: HTML-презентация и PDF-экспорт собраны.

Важно:
- На слайдах 6-7 пока заглушки для данных о выручке.
- PPTX-версия сохраняет редактируемый текст, поэтому градиенты упрощены.

Проверено: браузерный рендер, консоль и PNG всех слайдов через локальную копию
`node scripts/export_deck_png.mjs`.
```
