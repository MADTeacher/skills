# Оркестрация презентации через субагентов

Главный агент остается editor-in-chief: он держит бриф, принимает решения по истории, назначает bounded задачи, интегрирует результаты и отвечает за финальную сдачу. Субагенты нужны не для замены дизайнера, а для сохранения контекстной чистоты и параллельного выполнения независимых задач.

Используй этот reference только если пользователь или активная среда явно разрешили делегирование. Если разрешения нет, выполняй те же роли локально и последовательно.

## Когда делегировать

Делегирование особенно полезно, если:

- презентация длинная или многосекционная;
- нужно параллельно собрать брендовые и содержательные материалы;
- пользователь просит варианты направления;
- есть независимые блоки слайдов;
- нужен отдельный QA без подсказки ожидаемых дефектов.

Не делегируй, если задача крошечная, следующий шаг полностью заблокирован результатом делегируемой работы или write set неизбежно конфликтует.

## Роли

### 1. Research / asset scout

Задача:

- собрать недостающие материалы;
- пройти `brand-assets.md` или `content-assets.md`;
- зафиксировать пробелы, источники и риски.

Подходит для:

- логотипов, бренд-цветов, UI-скриншотов;
- картинок, фото, карт, диаграмм и предметных изображений;
- фактчекинга имен, чисел и обязательных утверждений.

Ожидаемый результат:

- список найденных файлов и путей;
- список пробелов;
- короткий `brand-spec.md` или notes-файл в рабочей директории презентации;
- без попытки финально переписать narrative всей презентации.

### 2. Direction worker

Задача:

- собрать 2-слайдовое направление: обложка + плотный контентный слайд;
- работать в отдельной ветке файлов или отдельной директории варианта.

Подходит для:

- 2-3 визуальных направлений;
- сравнения tonal systems;
- проверки, как narrative чувствуется в разных композициях.

Ожидаемый результат:

- готовые HTML-файлы варианта;
- список использованных токенов и принципов;
- короткое пояснение, для какой аудитории направление подходит лучше всего.

### 3. Slide production worker

Задача:

- собрать конкретный блок слайдов после утверждения направления.

Подходит для:

- секций с четкой ownership-моделью;
- приложений, appendix, data section, roadmap block;
- однотипных, но длинных серий слайдов.

Ожидаемый результат:

- изменения только в назначенных файлах;
- сохранение общих токенов и layout discipline;
- список измененных файлов и локально выполненных проверок.

### 4. QA / export reviewer

Задача:

- независимо открыть презентацию;
- пройти checklist из `html-decks.md`, `slide-qa.md`, `pptx-authoring.md` и `deck-review.md`;
- экспортировать PNG, вести `exports/slide-qa.md` и проверить слайды `1..N`, не получая от главного агента подсказок про ожидаемые баги.

Ожидаемый результат:

- findings с привязкой к конкретным слайдам или файлам;
- список выполненных проверок;
- список экспортированных артефактов;
- никаких молчаливых правок narrative без явного поручения.

## Правила handoff

Каждому субагенту обязательно передавать:

- цель задачи и ожидаемый артефакт;
- конкретную зону ответственности;
- допустимые файлы для записи;
- обязательные references;
- формат отчета назад: что изменено, какие проверки выполнены, какие риски остались.

Если есть несколько рабочих субагентов, явно пиши, что они не одни в кодовой базе и не должны откатывать чужие изменения.

## Шаблоны поручений

### Research / asset scout

```text
Используй skill по пути <path-to-skill>. Ты отвечаешь только за сбор материалов
для презентации <working-dir>. Пройди `references/brand-assets.md` или
`references/content-assets.md`, собери недостающие материалы, обнови только
файлы в <working-dir>/assets и notes-файлы в <working-dir>. Не меняй narrative и
не редактируй основные HTML-слайды. В конце верни список найденных файлов,
пробелы и выполненные проверки.
```

### Direction worker

```text
Используй skill по пути <path-to-skill>. Ты отвечаешь только за вариант
направления <variant-dir>. Собери 2-слайдовую связку: cover + dense content.
Соблюдай ограничения формата <target-format>. Не редактируй чужие варианты и не
трогай финальные файлы за пределами <variant-dir>. В конце перечисли измененные
файлы, дизайн-логику и ограничения.
```

### QA / export reviewer

```text
Используй skill по пути <path-to-skill>, чтобы проверить презентацию по
пользовательскому запросу: <user-request>. Не получай от меня список ожидаемых
ошибок. Самостоятельно открой локальные артефакты, пройди `html-decks.md`,
`slide-qa.md`, `pptx-authoring.md` и `deck-review.md`, выполни экспорт PNG,
обнови `exports/slide-qa.md` и верни findings, артефакты и blockers.
```

## Анти-паттерны

- Делегировать формулировку всей истории, а потом бездумно сшивать ответы.
- Отдавать нескольким субагентам один и тот же основной HTML-файл.
- Просить QA-агента "проверь именно баг X", а потом считать это независимой проверкой.
- Отправлять субагенту весь контекст проекта, если ему нужен только один reference и конкретная директория.
- Делать главного агента пассивным диспетчером без локальной интеграции.
